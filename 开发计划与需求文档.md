# 微信小程序诗词项目开发计划与需求文档

## 1. 项目概述

### 1.1 项目目标

本项目开发一个以诗词内容、飞花令互动、诗词浏览与个人中心为核心功能的微信小程序。前端使用原生微信小程序技术栈，后端使用 Python + FastAPI，数据库使用 SQLite，缓存层默认使用本地缓存，可按部署环境切换为 Redis。

小程序页面必须严格复刻 `template` 文件夹中的设计图片，包括布局、间距、颜色、字体层级、组件状态、图片比例和页面滚动效果。

### 1.2 技术栈

| 模块 | 技术选型 | 说明 |
| --- | --- | --- |
| 小程序前端 | 微信小程序原生框架 | 使用 WXML、WXSS、JavaScript/TypeScript、JSON |
| 后端 API | Python + FastAPI | 提供 RESTful API、认证、业务逻辑 |
| 数据库 | SQLite | 默认本地轻量数据库，便于开发和单机部署 |
| 缓存 | 本地缓存默认，Redis 可选 | 使用统一缓存工具封装，避免业务代码感知缓存实现 |
| 接口文档 | FastAPI OpenAPI | 自动生成接口文档，开发阶段辅助联调 |
| 数据校验 | Pydantic | 请求参数、响应数据统一建模 |
| ORM | SQLAlchemy / SQLModel | 推荐使用 ORM 管理 SQLite 表结构 |
| 运行管理 | uvicorn | FastAPI 本地开发和部署运行 |

### 2.1 UI 还原要求

1. 页面布局必须严格参考模板图片，不允许自行更改页面结构。
2. 颜色、字号、行高、圆角、阴影、间距、图标位置需要逐项对齐模板。
3. 页面首屏内容、底部导航、列表卡片、按钮状态必须与模板一致。
4. 长页面需要保证滚动后内容连续，不出现布局断层。
5. 图片素材、图标、背景图需要统一放入小程序 `assets` 目录。
6. 所有页面需要适配常见微信小程序机型宽度，优先使用 `rpx`。
7. 设计稿尺寸不统一时，以视觉比例为准，不按单一宽度强行拉伸。
8. 开发完成后需要对照模板截图验收，关键页面建议进行截图比对。

## 3. 用户角色与核心流程

### 3.1 用户角色

| 角色 | 说明 |
| --- | --- |
| 游客用户 | 未登录用户，可浏览基础内容，部分功能提示登录 |
| 登录用户 | 已完成微信登录，可收藏、参与飞花令、查看个人记录 |
| 管理员 | 可通过后端脚本或管理接口维护诗词数据、分类数据和推荐内容 |

### 3.2 核心用户流程

1. 用户打开小程序，进入登录页或首页。
2. 用户完成微信授权登录，后端创建或更新用户信息。
3. 用户在首页浏览推荐诗词、热门内容和功能入口。
4. 用户进入分类页，通过分类、作者、朝代或关键字查找诗词。
5. 用户进入诗词详情页，查看原文、译文、注释、赏析和相关信息。
6. 用户可收藏诗词，收藏结果同步到后端。
7. 用户进入飞花令页面，选择关键字或系统出题后进行答题。
8. 用户在广场页浏览公开内容、推荐内容或用户动态。
9. 用户在个人中心查看资料、收藏、历史记录和基础设置。

## 4. 功能需求

### 4.1 登录页

#### 页面目标

承接用户首次进入小程序时的身份确认，引导用户通过微信授权登录。

#### 功能点

1. 展示与 `template/登录页.png` 一致的背景、标题、按钮和页面层级。
2. 支持微信登录授权。
3. 获取用户基础信息，调用后端登录接口。
4. 登录成功后保存 `token`、用户 ID、昵称和头像。
5. 登录失败时给出轻量提示，不阻断用户返回。
6. 支持游客模式时，允许跳过登录进入首页，但收藏、飞花令记录等功能需要再次提示登录。

### 4.2 首页

#### 页面目标

作为小程序主入口，承载推荐内容、快捷功能入口和诗词内容导流。

#### 功能点

1. 严格复刻 `template/首页.png`。
2. 展示推荐诗词、精选分类、功能入口和热门内容。
3. 支持点击诗词进入详情页。
4. 支持点击分类或功能模块跳转对应页面。
5. 首页数据从后端接口获取，并使用缓存减少重复请求。
6. 下拉刷新时重新请求首页聚合数据。

### 4.3 广场页

#### 页面目标

展示公开内容流，可用于诗词推荐、用户动态、飞花令战绩或精选内容。

#### 功能点

1. 严格复刻 `template/广场.png`。
2. 支持内容列表分页加载。
3. 支持内容卡片点击进入详情页或动态详情页。
4. 支持点赞、收藏或分享入口，具体能力按模板中的可见组件实现。
5. 支持空状态、加载中、加载失败和无更多数据状态。

### 4.4 分类页

#### 页面目标

帮助用户按分类维度查找诗词。

#### 功能点

1. 严格复刻 `template/分类.png`。
2. 支持按朝代、作者、主题、体裁、关键字分类。
3. 支持分类列表和诗词列表联动。
4. 支持搜索关键词。
5. 支持分页加载分类下的诗词。
6. 点击诗词进入详情页。

### 4.5 飞花令页

#### 页面目标

提供飞花令玩法，让用户围绕关键字输入诗句并获得结果反馈。

#### 功能点

1. 严格复刻 `template/飞花令.png`。
2. 支持选择或随机生成飞花令关键字。
3. 支持用户输入诗句。
4. 后端校验诗句是否包含关键字，以及是否存在于诗词库。
5. 支持计分、答题记录和结果反馈。
6. 支持重新开始、换题、分享成绩。
7. 已登录用户保存飞花令记录，游客仅本地保存当次记录。

### 4.6 诗词详情页

#### 页面目标

展示诗词详细内容，提供阅读、收藏和学习信息。

#### 功能点

1. 严格复刻 `template/诗词详情.png`。
2. 展示标题、作者、朝代、正文、译文、注释、赏析。
3. 支持收藏和取消收藏。
4. 支持分享给微信好友或朋友圈。
5. 支持相关诗词推荐。
6. 用户访问详情后记录浏览历史。
7. 首次进入请求后端数据，重复进入优先使用缓存。

### 4.7 个人中心页

#### 页面目标

展示用户信息和个人数据入口。

#### 功能点

1. 严格复刻 `template/个人中心.png`。
2. 展示头像、昵称、登录状态。
3. 支持查看收藏列表。
4. 支持查看浏览历史。
5. 支持查看飞花令记录。
6. 支持清理本地缓存。
7. 支持退出登录。

## 5. 后端 API 需求

### 5.1 API 设计原则

1. 所有接口统一使用 `/api/v1` 前缀。
2. 响应结构统一，包含 `code`、`message`、`data`。
3. 业务异常使用 FastAPI 异常处理器统一转换。
4. 请求参数和响应结果必须使用 Pydantic 模型。
5. 接口函数需要写详细中文注释，说明入参、返回值和关键业务逻辑。
6. 公共工具逻辑必须封装到独立模块，禁止在路由层堆叠复杂代码。

### 5.2 基础响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 6.2 索引建议

1. `users.openid` 创建唯一索引。
2. `poems.title`、`poems.author`、`poems.dynasty` 创建普通索引。
3. `favorites.user_id + poem_id` 创建唯一联合索引，避免重复收藏。
4. `browse_history.user_id + viewed_at` 创建联合索引，提高个人历史查询性能。
5. `feihualing_records.user_id + created_at` 创建联合索引，提高记录查询性能。

## 7. 缓存设计

### 7.1 缓存目标

缓存用于减少重复读取 SQLite，提高首页、诗词详情、分类列表、飞花令关键字等高频数据的访问速度。

### 7.2 缓存实现原则

1. 默认使用本地内存缓存，适合开发环境和单机部署。
2. 通过配置切换 Redis，业务代码不直接调用 Redis。
3. 缓存能力必须封装到统一工具模块，例如 `app/core/cache.py`。
4. 缓存工具至少提供 `get`、`set`、`delete`、`clear_prefix` 方法。
5. 缓存 key 统一定义，避免路由层硬编码。
6. 写操作成功后，需要清理相关缓存。

### 7.3 推荐缓存 key

| Key | 说明 | 过期时间 |
| --- | --- | --- |
| `home:data` | 首页聚合数据 | 5 分钟 |
| `poem:detail:{id}` | 诗词详情 | 30 分钟 |
| `category:list` | 分类列表 | 30 分钟 |
| `category:poems:{id}:{page}` | 分类诗词列表 | 10 分钟 |
| `feihualing:keywords` | 飞花令关键字 | 30 分钟 |
| `square:feed:{page}` | 广场内容流 | 5 分钟 |

## 11. 开发阶段计划

### 第一阶段：项目初始化

1. 创建小程序工程目录。
2. 创建 FastAPI 后端工程目录。
3. 配置 SQLite 数据库连接。
4. 配置统一响应、异常处理、日志和基础配置。
5. 封装请求工具、缓存工具和鉴权工具。
6. 建立基础代码规范，保证中文注释和文件长度约束。

### 第二阶段：模板拆解与静态页面复刻

1. 按 7 张模板图拆分页面结构。

## 14. 风险与处理方案

| 风险 | 影响 | 处理方案 |
| --- | --- | --- |
| 模板尺寸不统一 | UI 还原难度增加 | 以视觉比例为准，逐页独立适配 |
| 微信登录依赖真实 AppID | 本地联调受限 | 提供 mock 登录配置，正式环境切换真实微信接口 |
| SQLite 并发能力有限 | 高并发场景受限 | 当前项目默认轻量部署，后续可平滑迁移到 MySQL/PostgreSQL |
| Redis 环境不一定存在 | 部署复杂度增加 | 默认本地缓存，Redis 作为可选增强 |
| 诗词数据质量不稳定 | 搜索和飞花令校验不准确 | 增加数据清洗脚本和人工校验字段 |
| 页面复刻细节多 | 验收周期变长 | 优先完成静态复刻，再接入动态数据 |
